# AI 双代理开发协作协议

- 项目：响帮帮无忧保
- 适用：Codex、Claude Code 及后续自动化代理
- 状态：生效
- 生效日期：2026-07-15

## 1. 目标

Codex 和 Claude Code 没有可靠的原生实时对话通道。本协议使用 Git 分支、独立工作树、任务交接文件和只读预检脚本完成异步协调，目标是避免文件覆盖、迁移分叉、权限泄漏、重复实现和相互部署。

协议优先级为：用户最新明确要求 > 已确认产品设计 > 本协议 > 任务交接文件 > 具体实现偏好。

## 2. 基本模型

- `main`：始终保持可构建、可回归、可发布，不直接开发功能。
- 一个任务：一个任务编号、一个主负责人、一个分支、一个工作树、一个交接文件。
- 一个公共模块：同一时间只有一个主负责人。
- 一个发布窗口：同一时间只有一个发布负责人。
- 没有声明和交接的修改不得合并。

建议工作树放在主仓库外，例如：

```text
/Users/madisonshen/Desktop/Demo
/Users/madisonshen/Desktop/Demo-worktrees/claude-<task>
/Users/madisonshen/Desktop/Demo-worktrees/codex-<task>
```

不要把新工作树创建在仓库的 `.claude/`、`.codex/` 或其他子目录内。已有工作树可完成当前任务，后续新任务改用外部目录。

## 3. 任务状态

交接文件只能使用以下状态：

- `planned`：已规划，尚未占用代码模块。
- `active`：正在开发，任务范围由负责人占用。
- `blocked`：因依赖或文件冲突暂停。
- `review`：开发完成，正在审查或补测试。
- `ready`：测试和交接齐全，可以进入合并队列。
- `merged`：已经合并到 `main`。
- `released`：已经发布并完成线上验证。
- `cancelled`：明确取消，不再占用模块。

只有 `active` 任务可以修改业务代码。`planned` 和 `blocked` 任务只能修改自己的设计、交接或完全不重叠的测试材料。

## 4. 启动前预检

每个代理开始或恢复任务时必须：

1. 更新本地对仓库状态的认识，不假设上次会话仍然有效。
2. 阅读本协议及所有未结束交接文件。
3. 运行 `bash scripts/ai_coordination_check.sh`。
4. 检查所有工作树是否干净、各分支基线及相对 `main` 的文件差异。
5. 在交接文件中声明计划修改的模块和迁移所有权。
6. 如果预检报告公共文件重叠或多个分支都有迁移，停止重叠实现，先确定依赖顺序。

预检是只读检查，不得自动重置、清理、暂存、提交、合并或删除任何文件。

## 5. 模块所有权与冲突处理

### 必须串行的公共区域

- 用户模型、登录、JWT、角色、租户和数据范围权限
- Alembic 迁移、启动兼容迁移和数据库初始化
- 参保、停保、生效规则、保费和有效天数计算
- 返佣、业务员佣金、平台使用费、充值和账本
- 首页、经营大屏、报表汇总
- `backend/app.py`、模型/Schema 聚合入口
- Web 公共类型、路由、导航和登录守卫
- 小程序全局认证、API 地址和上传版本
- Render 配置、生产环境变量和发布脚本

### 冲突决策

1. 生产紧急修复优先，但必须单独建 hotfix，并通知活动任务在合并后更新基线。
2. 非紧急任务中，先声明为 `active` 且范围合理的一方拥有模块。
3. 后开始的一方把相关工作标记为 `blocked`，可以先做不重叠文档、测试或页面。
4. 依赖任务必须从已合并的依赖结果继续；不要靠“合并时再解决”处理业务语义冲突。
5. 合并冲突不能机械选择 ours/theirs，必须按产品设计逐项确认数据结构、权限和计算口径。

任何代理不得编辑另一个代理的工作树来“帮忙修好”。需要修正时应提交到自己的分支，或在交接文件中把任务明确移交。

## 6. 数据库迁移锁

- 同一时间只有一个 `active` 任务持有 `migration_owner: yes`。
- 新迁移的 `down_revision` 必须指向最新已合并的迁移头。
- 如果另一个分支已有未合并迁移，新任务不得从旧 `main` 再创建迁移；应等待它合并，或明确以其分支为依赖基线。
- 已在任何共享或生产数据库运行的迁移不得改写，只能增加新的修正迁移。
- Python/Alembic 是唯一结构迁移来源；Java 实体、Mapper 和 SQL 必须跟随同一结构。
- 合并前必须验证空库初始化和旧库升级两条路径。

## 7. API 与前端协作

- 先确定请求、响应、权限、错误码和字段可见性，再并行实现前后端。
- 财务内部字段、其他业务员佣金、跨企业数据不得仅靠前端隐藏，必须由后端过滤。
- Web `types.ts`、路由和小程序全局文件视为公共接口，修改前必须确认所有者。
- 破坏性响应变更必须同步更新 Web、小程序、Java 镜像、测试和设计文档。

## 8. 本地运行隔离

推荐默认分配：

```text
Claude Code 后端：127.0.0.1:8001 + 独立测试数据库
Codex 后端：127.0.0.1:8010 + 独立测试数据库
合并后 main 联调：127.0.0.1:8000
```

双方不得共用可写数据库、同一开发端口或相同的小程序上传版本。上传、短信、邮件、支付和保司服务在分支测试时默认使用 mock。

## 9. 提交、测试与交接门槛

提交应保持小而完整，推荐顺序：设计/迁移、后端、前端、小程序、测试与文档。每次任务交接至少记录：

- 基线和分支
- 负责人及当前状态
- 范围和明确非范围
- 计划及实际修改文件
- 数据库迁移和兼容影响
- 提交编号
- 已运行测试及结果
- 未完成事项、已知风险和下一动作

进入 `ready` 前，按影响范围至少验证：

- Python 系统、安全和权限测试
- Java Maven 测试
- Web 正式构建
- 小程序语法检查、编译和预览
- 空库初始化、旧库升级和回滚方案
- 跨企业、跨实际工作单位、跨业务员的越权反向测试
- 导入模板与导出文件能够实际下载和打开

## 10. 合并规则

- 依赖任务先合并，后续任务基于更新后的 `main` 开发。
- 合并前检查 `main...branch` 全部差异，确认没有调试代码、凭据、本地数据库和构建临时文件。
- 合并后必须在 `main` 上重新运行受影响测试，不能只引用分支测试结果。
- 合并完成后把交接状态改为 `merged`，记录最终提交和剩余发布步骤。
- 不允许两个代理同时合并公共模块。

## 11. 发布安全闸门

开发授权不自动等同于生产发布授权。每次发布前必须确认该次发布范围，且只能有一名发布负责人执行：

- 推送发布分支或触发 Render 部署
- 执行生产数据库迁移
- 修改生产环境变量或密钥
- 上传微信小程序体验版
- 提交微信审核或正式发布

发布后验证健康检查、登录、权限、参停保、报表、导入导出和关键财务口径，再把交接状态改为 `released`。

## 12. 需要用户决策的情况

日常实现、测试和无歧义冲突由代理自行协调。只有以下情况升级给用户：

- 两项业务要求相互矛盾且无法从已确认设计判断优先级
- 删除、重算或不可逆迁移生产数据
- 真实资金、付款、生产密钥或外部账号操作
- 正式部署、微信提审或正式发布
- 两种实现都会改变合同、价格、保障或考核结果，且缺少业务依据
